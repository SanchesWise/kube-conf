Переход на Jumbo Frames (MTU 9000) — это обязательный шаг для 20-гигабитных линков, так как это значительно снижает нагрузку на процессоры (меньше прерываний на обработку пакетов) и увеличивает реальную пропускную способность для MinIO, iSCSI и NVMe-oF.

Важное правило: MTU должен быть настроен одинаково по всей цепочке: Физический порт -> Бонд -> Мост -> Коммутатор. Если хоть одно звено останется на 1500, пакеты будут дробиться (фрагментироваться) или отбрасываться, что приведет к падению скорости.

Шаг 1: Настройка MikroTik (CRS326)

В MikroTik есть два понятия: MTU (L3) и L2 MTU. Для Jumbo Frames нам нужно изменить оба.

Физические порты (SFP+): Для всех портов, куда воткнуты ноды (3, 4, 13, 14, 17, 18), установите:

MTU: 9000

L2 MTU: не менее 9000 (обычно по умолчанию на SFP+ он выше, например 9500 — это нормально).

Интерфейсы Bonding: Для всех созданных бондов (на 3 ноды) также установите:

MTU: 9000

Bridge: В разделе Bridge для того моста, в котором лежат бонды, установите:

MTU: 9000

Шаг 2: Настройка на нодах Proxmox

Нужно отредактировать /etc/network/interfaces на каждой ноде. MTU нужно прописать для физических интерфейсов, бонда и моста.

Пример для Node 1 (Gpu-prox):

code
Text
download
content_copy
expand_less
auto nic2
iface nic2 inet manual
    mtu 9000

auto nic3
iface nic3 inet manual
    mtu 9000

auto bond0
iface bond0 inet manual
    bond-slaves nic2 nic3
    bond-miimon 100
    bond-mode 802.3ad
    bond-xmit-hash-policy layer2+3
    mtu 9000

auto vmbr0
iface vmbr0 inet dhcp
    bridge-ports bond0
    bridge-stp off
    bridge-fd 0
    mtu 9000

Повторите это для всех нод, используя их имена интерфейсов (enp132... и т.д.).

Шаг 4: Важнейшая проверка (End-to-End)

После настройки на всех нодах и MikroTik проверьте, что пакеты реально проходят без фрагментации.

Запустите пинг между нодами с флагом запрета фрагментации:

code
Bash
download
content_copy
expand_less
# 8972 байт данных + 28 байт заголовков = 9000 байт
ping -s 8972 -M do 10.10.1.53

Если ответ пришел: Всё настроено идеально.

Если ошибка Frag needed and DF set: Где-то в цепочке (на роутере или порту) осталось MTU 1500.

Если пинг висит: Скорее всего, MikroTik или одна из нод не приняла настройки.

Шаг 5: Настройка внутри ВМ и Контейнеров

Чтобы ваш MinIO (в LXC) и Kubernetes (в VM) получили профит от 20 Гбит/с, их виртуальные адаптеры тоже должны знать про MTU 9000.

Для LXC (MinIO):
В Proxmox в разделе Network контейнера установите MTU: 9000.

Для VM (RedOS/K8s):
В Proxmox в настройках сетевого адаптера ВМ установите MTU: 9000.
Внутри самой RedOS также нужно поднять MTU:

code
Bash

content_copy
expand_less
# Для NetworkManager
nmcli connection modify "System eth0" 802-3-ethernet.mtu 9000
nmcli connection up "System eth0"

Ваше опасение абсолютно оправдано. Смешивать устройства с MTU 9000 и MTU 1500 в одном широковещательном домене (в одном бридже/подсети) без разделения на VLAN — это классический рецепт труднодиагностируемых проблем.

В чем именно риск?

Если сервер (MTU 9000) попытается отправить большой пакет устройству, которое понимает только 1500 (например, вашему компьютеру или принтеру в той же сети):

Пакет дойдет до устройства, но сетевая карта устройства его отбросит как гигантский/битый (Oversized Frame).

Пинг при этом будет работать (пакеты маленькие), а передача файлов или открытие веб-интерфейса будет «висеть» бесконечно.

Как сделать правильно и безопасно на MikroTik и Proxmox?

У вас есть два пути: «Безопасный L2» и «Профессиональный (VLAN)».

Вариант 1: Безопасный L2 (L2 MTU vs IP MTU)

На MikroTik есть два параметра. Секрет в том, чтобы разрешить «пропуск» больших пакетов на уровне железа, но не заставлять сам роутер использовать их для обычного трафика.

На MikroTik:

Для всех портов и самого Bridge установите L2 MTU в значение 9216 (максимально возможное). Это просто «расширяет трубу».

Actual MTU (обычный MTU) для Bridge оставьте 1500.

Это позволит пакетам 9000 проходить сквозь роутер между нодами Proxmox, но сам роутер и другие устройства будут общаться по 1500.

На нодах Proxmox:

Вы можете оставить vmbr0 на MTU 1500 (для управления и интернета).

Но это не даст вам 20 Гбит/с для MinIO и ZFS.

Вариант 2: Профессиональный (VLAN) — Рекомендуется для 20G

Раз у вас бонд 20 Гбит/с, лучше всего разделить трафик на уровне тегов.

VLAN 1 (Management): MTU 1500. Для интернета, управления Proxmox и обычных устройств.

VLAN 10 (Storage/Interlink): MTU 9000. Только для связи нод между собой, MinIO, iSCSI и миграции ВМ.

Как это настроить в Proxmox (/etc/network/interfaces):
Вместо одного vmbr0 мы делаем «VLAN-aware» мост:

code
Text
download
content_copy
expand_less
auto bond0
iface bond0 inet manual
    bond-slaves nic2 nic3
    bond-mode 802.3ad
    bond-xmit-hash-policy layer2+3
    mtu 9000  # На физике разрешаем максимум

auto vmbr0
iface vmbr0 inet manual
    bridge-ports bond0
    bridge-stp off
    bridge-fd 0
    bridge-vlan-aware yes
    mtu 9000 # Мост должен уметь пропускать 9000

# Интерфейс управления (VLAN 1 по умолчанию - MTU 1500)
auto vmbr0.1
iface vmbr0.1 inet dhcp
    mtu 1500

# Интерфейс для скоростного интерлинка (VLAN 10 - MTU 9000)
auto vmbr0.10
iface vmbr0.10 inet static
    address 10.10.10.55/24
    mtu 9000

На MikroTik:

Создайте VLAN 10 на интерфейсе Bonding.

Дайте VLAN 10 адрес 10.10.10.1 и MTU 9000.

Обычный бридж оставьте с MTU 1500.

Что если вы НЕ хотите возиться с VLAN? (Компромисс)

Если вы хотите оставить одну плоскую сеть 10.10.1.0/22:

На MikroTik: Поставьте L2 MTU 9216 на всё. IP MTU 1500 на Bridge.

На нодах Proxmox: Поставьте MTU 9000 на nic2, nic3, bond0.

Критический момент: На vmbr0 (где висит IP адрес сервера) ОСТАВЬТЕ MTU 1500.

Для MinIO и iSCSI: Внутри Proxmox создайте дополнительные виртуальные интерфейсы или используйте настройки внутри контейнеров/ВМ, указывая там MTU 9000.

Но внимание: В плоской сети vmbr0 с MTU 1500 не даст вам преимуществ Jumbo Frames для трафика самого хоста (например, при репликации ZFS).

Итоговый совет:

Если вы не планируете настраивать VLAN, не ставьте MTU 9000 на vmbr0. Поставьте его только на физические порты и бонд (на уровне железа/L2).

Это позволит вам позже «прокинуть» MTU 9000 внутрь конкретных ВМ для Kubernetes или MinIO, не ломая доступ к самому Proxmox для обычных компьютеров с MTU 1500.

Для проверки на MikroTik выполните:
/interface print detail — посмотрите колонку L2 MTU. Если там везде > 9000, то пакеты 9000 между нодами пролетят, даже если сам мост роутера настроен на 1500.
